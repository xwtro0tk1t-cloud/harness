# 审计方法论

生成的 security-review-skill 应指导 Claude 遵循以下审计方法。

## 核心原则

**反幻觉原则**：宁可漏过漏洞，也不误报
- 所有发现必须基于实际代码读取
- 禁止基于"典型项目结构"猜测
- 文件路径和代码片段必须来自实际输出

## 1. 侦察阶段（审计前必做）

**先理解全局，再开始审计：**

### 技术栈识别
- 识别语言、框架、数据库
- 确定适用的漏洞模式和检查项

### 攻击面映射
- 枚举所有 API 端点（routes, controllers）
- 识别入口点：main、handlers、webhooks
- 定位关键模块：auth、payment、api、admin

### 端点-权限矩阵
生成端点与权限要求的对照表，供后续授权检查：
```
| 端点 | 方法 | 需要认证 | 需要角色 | 实际检查 |
|-----|------|---------|---------|---------|
| /api/user/{id} | GET | 是 | user | ? |
| /api/admin/users | GET | 是 | admin | ? |
```

### 组件交互分析
- 梳理调用链：API → Service → DAO → DB
- 关注跨组件的数据传递

### 外部依赖分析
- 检查 import/require 引入的第三方库
- 识别高风险依赖（序列化、模板、执行类）
- 注意依赖版本是否有已知漏洞

## 2. 双轨审计模型

**两种互补的审计策略：**

### Sink-driven（汇点驱动）
**找"存在的危险代码"**

适用于：注入、反序列化、文件操作、SSRF、RCE

流程：
1. 搜索危险函数模式（execute, system, render, open...）
2. 追踪数据流：Source → Propagation → Sink
3. 验证是否有有效防御

### Control-driven（控制驱动）
**找"不存在的安全控制"**

适用于：认证、授权、业务逻辑漏洞

流程：
1. 枚举所有端点（从侦察阶段的端点-权限矩阵）
2. 验证每个端点是否有应有的安全控制
3. **缺失即脆弱**：应该有认证但没有 → 漏洞

示例：
```
端点: POST /api/user/{id}/delete
应有: @require_auth + @require_role('admin') + owner_check
实际: 只有 @require_auth，无角色检查
判定: 纵向越权漏洞
```

## 3. 渐进式分析

**先广后深，分两轮：**

### 第一轮：快速扫描
- Sink-driven：搜索危险函数，标记可疑点
- Control-driven：检查端点权限控制
- 不深究，先列出所有潜在问题

### 第二轮：深度验证
- 对每个可疑点追踪数据流
- 验证安全控制的有效性
- 判定：真实漏洞 / 误报 / 需确认

## 4. 数据流分析

**追踪数据的完整生命周期：**

```
Import (外部引入)
    ↓
Source (输入源) → Propagation (传播) → Sink (危险操作)
```

### 关注点

| 阶段 | 关注内容 |
|-----|---------|
| **Import** | 第三方库风险、不安全的反序列化库、已知漏洞组件 |
| **Source** | request 参数、文件读取、数据库查询结果、环境变量 |
| **Propagation** | 字符串拼接、赋值传递、函数调用、数据结构操作 |
| **Sink** | execute、system、render、write、网络请求 |

### 分析问题
1. 数据**从哪来**？（用户可控？外部依赖？）
2. 中间**经过什么处理**？（验证、过滤、转义？）
3. 最终**到达哪里**？（危险函数？）
4. **能被外部触发吗**？（有认证？内部接口？）

## 5. 跨文件分析

**遇到 wrapper 函数时，要查调用者：**

```python
# 这个函数本身"看起来"危险
def execute_query(sql):
    cursor.execute(sql)
```

**不要直接报漏洞！** 检查：
- 搜索所有调用 `execute_query()` 的地方
- 分析每个调用者传入的参数来源
- 只有当调用者传入用户可控数据时才是真正的漏洞

## 6. Git Diff 分析

**针对代码变更的增量审计：**

### 变更分析
- 识别本次变更修改了哪些文件
- 重点关注安全敏感文件的变更（auth、payment、api）
- 分析变更引入了什么新逻辑

### 变更上下文
- 变更涉及的函数被谁调用？
- 变更影响的数据流路径
- 变更是否破坏了原有的安全检查？

### 历史漏洞模式
- 检查该文件/模块是否有历史漏洞
- 类似的变更模式是否曾导致问题

## 7. 误报消除

**以下情况通常是误报：**

| 情况 | 示例 |
|-----|------|
| ORM 安全使用 | `objects.filter()`, `query().filter()` |
| 参数化查询 | `cursor.execute(sql, params)` |
| 框架自动转义 | Jinja2 autoescape, React |
| 输入已验证 | 白名单、类型转换、正则校验 |
| 内部调用 | 参数来自内部代码，非用户输入 |
| 测试代码 | tests/ 目录 |
| **不可外部触发** | 内部接口、需管理员权限、定时任务 |

### 可达性验证
判断漏洞是否可被外部触发：
- 追踪调用链，能否从公开接口到达？
- 中间是否有认证/授权检查？
- 是否只能内部调用？

## 8. 攻防对抗验证

**对可疑漏洞进行攻防验证：**

### 攻击方视角
自问："如果我是攻击者，如何利用这个漏洞？"
- 构造具体的攻击输入
- 描述攻击步骤和路径
- 评估成功利用的条件

### 防御方视角
自问："有什么阻止这个攻击成功？"
- 是否有输入验证/过滤？
- 是否有框架保护？
- 是否有权限控制？
- 是否是内部接口？

### 最终判定
| 攻击方 | 防御方 | 判定 |
|-------|-------|------|
| 有明确利用路径 | 无有效防护 | **真实漏洞** |
| 有利用可能 | 有部分防护 | **需人工确认** |
| 利用条件苛刻 | 有有效防护 | **低风险/误报** |

## 9. 审计优先级

**优先检查：**
- 认证/授权：login, auth, permission, session
- 支付/金融：payment, checkout, balance, refund
- API 入口：routes, controllers, handlers
- 数据访问：models, dao, queries, repository

**可跳过：**
- 纯 UI 渲染
- 测试代码
- 静态配置文件

## 10. 覆盖评估

**审计完成前，检查覆盖率：**

### 关键维度检查
确保以下维度都已覆盖：

| 维度 | 检查项 | 是否覆盖 |
|-----|--------|---------|
| 注入 | SQL/命令/模板/反序列化 | ? |
| 认证 | 登录/会话/Token | ? |
| 授权 | 端点权限/资源归属 | ? |
| 文件 | 上传/下载/路径遍历 | ? |
| 业务逻辑 | 状态机/竞态/价格 | ? |

### 补充审计
如果发现某维度未覆盖：
- 返回该维度进行补充检查
- 特别是：认证 + 授权 + 注入 必须覆盖

